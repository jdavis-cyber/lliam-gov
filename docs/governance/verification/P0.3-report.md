## Objective
Disable the offensive/jailbreak skills and enforce a skill enablement gate.

## Key finding
`agent.skill_utils.get_disabled_skill_names()` reads the home `config.yaml` directly and **never sees the `HERMES_CONFIG` overlay**. The gate therefore reads the policy file + overlay-aware effective config independently and unions in, so godmode is denied regardless of where it is declared.

## What was tested
1. Low-side baseline (no overlay): godmode not disabled (gate inactive).
2. Under the gov overlay: godmode refused at the registration chokepoint; an allowlisted skill still loads.
3. Startup enablement gate logs a clear denial reason.
4. Intent-tag (`deny_tags`) denial of a re-vendored offensive skill.
5. Native `get_disabled_skill_names` behavior preserved (regression).

## Results (actual)
- Baseline (no overlay): `get_disabled_skill_names()` does **not** include godmode — gate inactive, low-side untouched.
- Gov overlay: disabled set = `[godmode, obliteratus, oss-forensics, sherlock, web-pentest, freedommode]`; slash-command map surfaced **only `pdf`** → `/godmode` **refused**, `/pdf` registrable.
- Startup gate log: `DENY skill 'godmode' — named in least-functionality denylist (mandatory must-disable) (LG-SC-01)`.
- Intent-tag: a re-vendored jailbreak skill (`freedommode`, tagged `jailbreak/uncensoring`) **refused** at registration via cached deny-tag scan.
- Regression: native `disabled` + `platform_disabled` honored (`{foo}` global, `{foo,bar}` for discord) → **PASS**.
- Capability: agent ran a coding task cleanly under the overlay.

## Expected-deny / expected-block cases
- `godmode`, `obliteratus`, `web-pentest`, `sherlock`, `oss-forensics` → **denied** (offensive set).
- Re-vendored skill matching `deny_tags` → **denied**.
- Non-allowlisted but benign skill → **detect-and-log only** (capability-preserving; not blocked).

## Capability preservation
Graduated: offensive/jailbreak = hard-deny; merely-not-on-allowlist = logged, not blocked (the ~120 built-in skills keep working).
