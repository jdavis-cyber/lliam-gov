## Objective
Pin the dependency supply chain (upper bounds + hashes) and disable runtime lazy installs for the CUI/air-gap profile.

## What was tested
1. The lockfile + hashes are committed (LG-SS-01).
2. A runtime lazy-install attempt is refused under the gov profile (LG-SS-07).

## Results (actual)
- **Pinning evidence (LG-SS-01):** `pyproject.toml` and `uv.lock` both git-tracked on the phase branch. `uv.lock` carries **1798 `sha256` hashes**. `pyproject.toml` `[project].dependencies` use exact pins (`==X.Y.Z`) / bounded ranges (`>=floor,<next_major`). The existing `.github/workflows/supply-chain-audit.yml` "Check PyPI dependency upper bounds" job fails the build on any unbounded dep. Even the optional-backend `LAZY_DEPS` specs are version-pinned.
- **Lazy-install refusal (LG-SS-07):**
  - Baseline (no overlay): `_allow_lazy_installs() = True` (shipped default).
  - Gov overlay: `security.allow_lazy_installs = False`, `security.require_hashed_install = True`, `_allow_lazy_installs() = False`.
  - Real attempt: `tools/lazy_deps.ensure("search.exa")` (missing dep `exa-py==2.10.2`) → **REFUSED** with `FeatureUnavailable: "lazy installs disabled (security.allow_lazy_installs=false)"` — raised **before** any pip call.

## Expected-deny / expected-block cases
- Any runtime backend self-install in the gov profile → **refused** (`FeatureUnavailable`).

## Capability preservation
Low impact, preserved by construction. No runtime agent capability removed; pre-provisioned backends work normally. The lazy-install kill switch has been active since P0.1 and the gov profile ran end-to-end in the P0.4/P0.5 capability smokes; P0.6 added only declarative supply-chain keys with no runtime consumer. Only the self-install of a *not-pre-provisioned* optional backend is blocked (air-gap posture; add it to the pinned lock + pre-provision instead).
